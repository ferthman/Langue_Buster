This audit treats the ZIP as a visual/frontend prototype only. Repository docs in `README.md`, `plan.md`, and `AGENTS.md` remain the source of truth.

## What The ZIP Contained

- A standalone React/Vite app with a single-screen game flow and local Telegram helpers.
- Prototype components for `Game`, `GameBoard`, `QuestionCard`, `ShapeTray`, `TopBar`, `WelcomeScreen`, and `PauseMenu`.
- Local board/question logic under `src/app/utils/*`.
- Local vocabulary and shape data under `src/app/data/*`.
- A large autogenerated `src/app/components/ui/*` surface from a Figma-style export.
- Imported raster assets under `src/imports/*`.

## Repository State Before Integration

- Monorepo boundaries already existed and matched the intended architecture:
  - `apps/miniapp` for Telegram frontend
  - `apps/api` for run/session/backend logic
  - `packages/game-engine` for deterministic board rules
  - `packages/content-core` for question/content logic
  - `packages/shared` for contracts
- `packages/game-engine` already had pure board placement, line clear, score, combo, and legal-move logic with tests.
- `packages/content-core` already had Phase 6 question generation and answer evaluation with strict distractor validation.
- `apps/miniapp` already used server-backed run state instead of client-authored game truth.

## Reused From The ZIP

- Layout composition ideas only:
  - strong separation of top bar, question card, board, tray, and summary feedback
  - compact mobile-first stacking for the gameplay screen
  - clearer tray-as-selection-zone affordance
- A few interaction ideas in adapted form:
  - immediate post-answer feedback before placement
  - visibly separate board and tray zones
  - stronger end-of-turn move summary presence

## Rewritten Instead Of Reused

- All gameplay logic from the prototype.
  - The ZIP mixed board logic, score, hearts, and question flow directly into React state.
  - The real project already had better pure-domain implementations.
- All question generation and weak-word logic from the prototype.
  - The ZIP used probabilistic resurfacing and random question selection.
  - Integration replaced this with deterministic short-cycle recovery in domain/backend layers.
- Telegram integration details.
  - The ZIP helper was a minimal browser wrapper.
  - The repo already had a proper Telegram provider with theme and haptics integration.
- All onboarding and product copy.
  - Prototype text was partially French/English and not aligned with MVP Russian UI requirements.

## Rejected Or Quarantined

- `src/app/components/ui/*`
  - autogenerated `shadcn`-style surface not needed for the real app
  - too broad for the actual Telegram Mini App scope
- `src/imports/image.png`
  - highly clone-like board screenshot with glossy blocks and commercial-puzzle resemblance
- `src/imports/image-1.png`
  - settings art visually tied to a commercial puzzle-game pattern
- `src/imports/image-2.png`
  - direct clone-risk branding and menu composition
- Prototype gradients, trophy/crown framing, and arcade-menu styling
  - too close to Block Blast / adjacent commercial puzzle identity territory

## Major Risks Found

- Legal/design risk:
  - imported artwork and overall visual language were too close to commercial block-puzzle products
- Architecture risk:
  - game and learning logic lived inside React components
- Product risk:
  - hearts hardcoded to 3 in UI
  - text not normalized to Russian MVP UI
  - weak-word handling based on random chance instead of deterministic resurfacing
- Maintenance risk:
  - large autogenerated component surface with low signal for this codebase

## Decisions Made

1. Preserve repo architecture and do not transplant the ZIP app into the monorepo.
2. Keep gameplay rules in `packages/game-engine`, not in React.
3. Keep question/content logic in `packages/content-core`, not in `apps/miniapp`.
4. Add deterministic short-cycle recovery support through domain logic and backend run selection.
5. Centralize current run defaults through shared constants instead of UI literals.
6. Rebuild the Mini App run screen using safe layout inspiration only, with Russian UI copy and no risky assets.

## Integrated Changes

- Added shared classic-run constants for board, tray, hearts, and short-cycle gap.
- Added shared run recovery state contract.
- Added `packages/content-core/src/short-cycle-recovery.ts` with deterministic recovery scheduling and source-item selection.
- Updated `apps/api` run service to:
  - default to 5 hearts
  - persist recovery state per run
  - schedule wrong answers into short-cycle recovery
  - skip invalid question sources instead of crashing run generation
- Reworked `apps/miniapp` run UI into the canonical order:
  - header
  - question card
  - board
  - tray
  - move summary
- Kept prototype assets out of the product.
